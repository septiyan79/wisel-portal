# Architecture

## Stack

- **Next.js 16.2.1** (App Router, Turbopack) — note: `AGENTS.md` warns this Next.js version has breaking changes vs. training-data knowledge; check `node_modules/next/dist/docs/` before relying on framework conventions you "remember."
- **TypeScript**, **React 19.2.4**
- **Prisma 5.22.0** + **PostgreSQL** (Neon), connection via `DATABASE_URL`/`DIRECT_URL`
- **NextAuth v5 (beta)** — Credentials provider, JWT session strategy
- **Tailwind CSS v4** + **shadcn/ui** (Radix primitives) — see caveat in [known-issues.md](known-issues.md) about the shared `Button` component being unused by the dashboard layer
- **bcryptjs** — password hashing
- **xlsx** — Excel import/template generation (client and server)
- **googleapis** — Google Sheets export (service account)
- **ai** SDK + **@ai-sdk/groq** — AI chat, Groq-hosted `llama-3.3-70b-versatile`
- **zod** — input validation (chat tool schemas; not used for API route body validation)

## Auth flow

1. **`proxy.ts`** (NextAuth middleware, matches all routes except `/api`, static assets): redirects unauthenticated users away from `/dashboard`, `/transactions`, `/units`, `/profile`; redirects authenticated users away from `/login`. This is a coarse, path-prefix-based gate — it does **not** cover `/api-keys`, `/users`, `/docs` by prefix match (they don't start with the listed prefixes), so those rely entirely on the `(dashboard)/layout.tsx` guard + per-page role checks below.
2. **`app/(dashboard)/layout.tsx`**: `const session = await auth(); if (!session) redirect("/login")`. This is the real choke point for every route under the `(dashboard)` group regardless of path prefix. It does **not** check role — only "is there a session."
3. **Per-page role gating**: each page under `(dashboard)` does its own role check. See [features.md](features.md) for the full table — there are two distinct strategies in use (hard redirect vs. scoped Prisma `where`), described below.
4. **Session shape** (`types/next-auth.d.ts`, populated in `lib/auth.ts` JWT/session callbacks): `{ id, role, customerAccount, customerName }`. `role` is a free-form string (`"customer"` or `"admin"` by convention, not a DB/TS enum — nothing stops a third value existing except application logic).
5. **API key auth** (separate path, only for `/api/customer/items`): `Authorization: Bearer wsl_<64 hex chars>` checked against `ApiKey.key` via `lib/api-auth.ts`. No session/cookie involved. Keys are generated by admins via `/api/admin/api-keys` and shown once (never retrievable again).

## Role-scoping conventions (read this before adding a new page/route)

Two patterns coexist in the codebase — know which one you're extending:

1. **Hard redirect gate** — page is entirely inaccessible to a role: `if (session.user.role === "customer") redirect(...)`. Used by `/api-keys`, `/units` (customer → `/transactions`), `/users` (stricter: `role !== "admin"` → `/dashboard`).
2. **Same page, scoped data via conditional Prisma `where`** — both roles see the same UI, but the query is scoped: `role === "customer" ? { ...filter, customerAccount: session.user.customerAccount } : { ...filter }`. Used by all `/transactions/*` pages and the shared `TransactionKpiCards` component. On detail pages (`by-fleet/[fleet]`, `by-part/[part]`), a customer requesting another customer's resource gets an **implicit 404** (empty scoped query result → `notFound()`) rather than an explicit 403 — this is intentional-looking but means "not found" and "not yours" are indistinguishable to the user.

When adding a new admin-only route, prefer being explicit about which of these two patterns you're using, and see [known-issues.md](known-issues.md) for the `!== "customer"` vs `!== "admin"` inconsistency before picking a check.

## Data flow pattern

- **Reads**: dashboard pages are async Server Components that query Prisma directly (scoped per the rules above) and pass the results as props into a client "Tab" component (`OrdersTab`, `StockTab`, `UnitsTab`, `UsersTab`, `ApiKeysTab`). Several of these client components seed local state from props via `useState(initialProp)` **without** a `useEffect` resync — meaning `router.refresh()` re-renders the server parent with fresh data, but the client child's local copy only updates for mutations the child itself performs (see [known-issues.md](known-issues.md)).
- **Writes**: client components call the internal REST API under `app/api/**` (fetch), then either call `router.refresh()` (re-runs the server component tree) or optimistically patch local state — the two are used inconsistently across components.
- **External sync**: `POST /api/sync` (no auth) accepts a full snapshot of provider transactions, upserts by `externalId`, and soft-deletes any `source: "provider"` row missing from the payload. This assumes each call is a *complete* snapshot across all customers.
- **Excel import**: `/api/transactions/import` and `/api/units/import` parse an uploaded workbook server-side with `xlsx`, row-by-row create/upsert, and return a `{ success, errors, total }` report. Transaction import requires the referenced `Unit` to already exist; units import upserts.
- **Export**: `lib/gsheets.ts` `exportToSheets()` is fired (`void`, fire-and-forget) after nearly every `Transaction`/`StockAssignment` mutation — it does a full re-read of all non-deleted transactions and overwrites a Google Sheet tab named `"export"`. It never mirrors `Unit`/`User`/`ApiKey` changes.

## AI Chat feature

Mounted globally via `ChatBubble` in `(dashboard)/layout.tsx` (every authenticated route, not the public landing page). Flow: `ChatBubble` (client, holds message history in React state only — no persistence) → `POST /api/chat` with the full message array → `generateText` (Groq `llama-3.3-70b-versatile`, non-streaming, max 5 tool-call steps) → tool-calling into `lib/chat-tools.ts` Prisma helpers (`queryTransactions`, `getSummaryStats`, `getStockInfo`, `searchParts`).

**Security-relevant detail**: the route computes `scopedAccount = role === 'customer' ? customerAccount : null` from the session *before* any model interaction, and every tool's `execute` forcibly overrides `customerAccount` with this server-side value — the model's tool-call arguments never include a `customerAccount` field (not in the Zod input schemas), so prompt injection cannot escalate a customer's chat session to see another customer's data. Admins get `scopedAccount = null` → unscoped, sees all customers by design.

## Magic strings / constants worth knowing globally

- `"WSL-000039232"` — hardcoded placeholder `Unit.deviceNumber` representing generic warehouse stock. Auto-assigned to Stock-category (`"S"`) import rows with no explicit device. Cannot be edited or deleted via `/api/units/[id]`.
- `"STOCK"` — a *different* sentinel device string, blocked specifically in `/api/stock-assignments` as an invalid `targetDeviceNumber` (you can't assign stock back to "stock"). Don't confuse with the placeholder unit above.
- Category codes: `"P"` = PM, `"R"` = Repair, `"S"` = Stock. Stored as single letters in the DB; import accepts full words (`PM`/`Repair`/`Stock`, case-insensitive) and normalizes them — see [known-issues.md](known-issues.md) for the historical bug here (fixed 2026-07-05).
- `source` field on `Transaction`: `"manual"` (created via the app forms — the only kind editable/deletable via `/api/transactions/[id]`), `"import"` (Excel), `"provider"` (external sync). A fourth value, `"stock_assignment"`, appears only in the **in-memory merged list** built by `transactions/summary/page.tsx` (StockAssignment rows displayed alongside Transactions) — it is never actually stored in the `Transaction.source` column.
