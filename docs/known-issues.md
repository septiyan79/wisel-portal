# Known Issues & Inconsistencies

Cross-cutting quirks discovered while documenting the codebase (2026-07-05). None of these block normal feature work, but check here before "fixing" something that turns out to be a pre-existing, deliberate-looking pattern — or before copying a pattern that's actually a bug.

## Fixed

- **Self-service password change + a real "forgot password" page** (fixed 2026-07-05): `ProfileTab`'s "Change Password" form used to be entirely fake — no fetch call, just a local 3-second "saved" flag, and there was no password-change API route at all. Since this portal has no email/phone on `User` (login is by `username`, not email — see [database.md](database.md)), a self-service *reset* flow (emailed reset link, OTP) isn't viable. Implemented instead: `PATCH /api/profile/password` (session-scoped to `session.user.id`, requires `currentPassword` to match via `bcrypt.compare` before setting `newPassword`, min 6 chars — no admin privilege needed since it can only ever target the caller's own row) wired up to `ProfileTab`'s form. The login page's "Forgot password?" link (`app/(auth)/login/page.tsx`) previously pointed at `/forgot-password`, a route that didn't exist (404) — added `app/(auth)/forgot-password/page.tsx`, a static page instructing the user to contact their admin, who can already reset any user's password via `PATCH /api/admin/users/[id]`.
- **Admin tenant leaked into customer picker dropdowns** (fixed 2026-07-05): `GET /api/customers` used to return every `Customer` including the special admin-only tenant (conventionally `customerAccount: "ADMIN"`, created just to satisfy the `User.customerAccount` FK), so it showed up as a selectable "customer" in `OrdersTab`'s filter, `TransactionFormModal`/`ImportModal`'s "on behalf of" picker, and `UsersTab`'s Account ID dropdown — none of which should ever target it. Fixed by excluding any `Customer` that has a linked `role: "admin"` user (`where: { users: { none: { role: "admin" } } }`) rather than hardcoding the `"ADMIN"` string, so it keeps working even if the admin tenant is ever recreated under a different Account ID. `GET /api/admin/customers` (used by `CustomersTab`) is intentionally unaffected — admins still need to see/manage that tenant there.
- **Dedicated Customer Management page + `customerName` editing removed from `UsersTab`** (fixed 2026-07-05): previously, `UsersTab`'s Add/Edit User form both created tenants on the fly (typing a new `customerAccount` + `customerName` upserted a `Customer`) and let *any* login under a shared `customerAccount` rename that tenant's `customerName` for every sibling login — the latter had only been mitigated with an inline warning, not actually fixed. Added `/customers` (`CustomersTab`) + `GET/POST /api/admin/customers` + `PATCH/DELETE /api/admin/customers/[id]` as the single place to create/rename/delete tenants (delete blocked if the customer has any linked users/units/transactions/API keys). `UsersTab`'s "Account ID" field is now a `<select>` of existing customers only (fetched from `GET /api/customers`) with "Company Name" auto-filled and permanently read-only, and `POST /api/admin/users` now 404s if the given `customerAccount` doesn't already exist instead of upserting one.
- **Admin accounts can't be deleted, duplicated, or demoted** (fixed 2026-07-05): `role: "admin"` is capped at one account system-wide — `POST`/`PATCH /api/admin/users[/id]` 409 if you try to create or promote a second one — and `DELETE /api/admin/users/[id]` rejects deleting any `role: "admin"` user outright (400 `"Admin accounts cannot be deleted"`), not just self-deletion. **A second gap in the same family was found and fixed after initial release**: `PATCH /api/admin/users/[id]` only blocked *promoting* a second admin, not *demoting* the existing one — editing the admin user's role to `"customer"`/`"customer_user"` via `UsersTab` would have silently succeeded and left the system with zero admins, with no way back in (worse than the delete case, since nothing short of a direct DB edit could recover). Now blocked server-side (400 `"The admin account's role cannot be changed"`) whenever `user.role === "admin"` and a different role is requested. This closes what would otherwise be a lockout: since only an existing admin can promote someone else to admin, losing the sole admin (via delete *or* role change) would have left no one able to reach `/users`, `/api-keys`, `/units`, `/customers`, or any admin-only API route. `UsersTab` mirrors this: delete button hidden for admin rows, "Admin" role option disabled once another admin exists, and *all three* role buttons disabled/locked when editing the admin account itself.
- **Multi-account-per-customer + role-check strictness** (fixed 2026-07-05): added a third role, `"customer_user"`, so a single `Customer` tenant can have more than one login (`User.customerAccount` is no longer unique — see [database.md](database.md), migration `20260705010000_multi_login_per_customer`; `User.username` is now the unique login identity). Introducing this third role activated the exact risk this doc previously warned about: every route/page that checked `role === "customer"` (or `!== "customer"`) would have silently treated `"customer_user"` as admin-equivalent. Audited and normalized **every** such check across ~20 files (pages, API routes, components) to `role !== "admin"` / `role === "admin"` instead — see [architecture.md](architecture.md#role-scoping-conventions). Also fixed a related latent bug surfaced by the same change: `DELETE /api/admin/users/[id]`'s self-deletion guard compared `user.customerAccount === session.user.customerAccount`, which would have incorrectly blocked deleting a *different* login that happens to share a tenant; it now compares `User.id`.
- **Excel import category mismatch** (fixed 2026-07-05, commit `44eac4e`): `/api/transactions/import` used to uppercase the Category cell verbatim, so a full word like `"Repair"` was stored as `"REPAIR"` instead of the single-letter code `"R"` that `TransactionKpiCards` and the Stock page filter on. Rows imported "successfully" but were invisible to KPI cards/Stock view even after a hard refresh. Fixed by adding a `CATEGORY_MAP` normalization (`PM/P→P`, `REPAIR/R→R`, `STOCK/S→S`) plus a rejection error for unrecognized values; the downloadable template was also updated to show all three category words. Data already imported with the wrong category had to be corrected manually in the DB.
- **`Unit.customerAccount` was nullable and unpopulated for every existing row** (fixed 2026-07-05): all 145 units in the DB had `customerAccount = null` — the field existed but nothing ever wrote to it (the units Excel import didn't even have a column for it). This meant `GET /api/units` returning an unfiltered list to every role was effectively unavoidable at the time (there was no data to filter by), and business intent ("every unit belongs to exactly one customer, and can move to another one") wasn't enforced anywhere. Fixed by: (1) backfilling all existing units to their actual owning customer, (2) making `Unit.customerAccount` `NOT NULL` in the schema (migration `20260705000000_make_unit_customer_account_required`), (3) requiring and validating `customerAccount` in `POST`/`PATCH /api/units[/id]` and in `/api/units/import` (new `Customer Account` column), (4) scoping `GET /api/units` by `customerAccount` for `role === "customer"`, and (5) removing the "None / Not assigned" option from `UnitFormModal`'s Customer select. Reassigning a unit (via the edit form or re-import) does not retroactively change historical `Transaction`/`StockAssignment` records, since those store their own `customerAccount` independently — this was verified to already work correctly.
- **`UnitFormModal` pre-selected the owning customer by matching `customerName` string** (fixed 2026-07-05): now `units/page.tsx` selects `Unit.customerAccount` directly and `UnitFormModal` uses it as-is, instead of doing a lossy `customers.find(c => c.customerName === initial.customerName)` lookup.
- **No cross-check between a transaction's `customerAccount` and the `Unit.customerAccount` of the device it's attached to** (fixed 2026-07-05): `POST/PATCH /api/transactions`, `POST /api/transactions/import`, and `POST/PATCH /api/stock-assignments` all let a `deviceNumber`/`targetDeviceNumber` be attached to a transaction or assignment belonging to a *different* customer than the one who owns that unit, with zero validation (manual create didn't even check the unit existed — it relied on an unhandled DB foreign-key error). Fixed by adding `lib/unit-validation.ts` (`validateUnitOwnership(deviceNumber, expectedCustomerAccount)`) and calling it from all four write paths, returning a friendly `400` instead of silently mismatched data or an unhandled FK error. **Exception**: the system stock placeholder `WSL-000039232` is excluded from the ownership check in `/api/transactions/import` (see the magic-strings note below) since it's a shared bucket, not really owned by the single customer its `customerAccount` happens to point to.

## Deferred (deliberately not doing these — decided 2026-07-05)

- **No audit trail for `Unit.customerAccount` reassignment**: when a unit moves from one customer to another (via `PATCH /api/units/[id]` or re-importing `/api/units/import` with a different `Customer Account`), the previous owner is simply overwritten — there is no history table recording who owned it before, when it changed, or who made the change. Would require a new model (e.g. `UnitOwnershipHistory`) written to on every reassignment. Explicitly deferred, not a bug — revisit if/when a "who owned this unit and when" report is actually needed.
- **`StockAssignment` has no `customerAccount` column of its own**: ownership is derived via a join to `stockTransaction.customerAccount` (who purchased the stock), not stored directly on the assignment row. This was investigated and confirmed **not a bug** — the join-based attribution is already correct everywhere it's used (verified against `by-fleet` scoping). Adding a denormalized column would only be a query-convenience/performance optimization for future reporting, not a correctness fix. Explicitly deferred — revisit only if a reporting need makes the join genuinely costly.

## Open / unfixed

- **`StockTab.tsx` doesn't resync from server props**: `useState<StockRow[]>(initialTransactions)` seeds once on mount with no `useEffect` to follow prop changes. If the underlying data changes (e.g. an import on another tab) and the user navigates back without a full page reload, the Stock page's table/KPI can show stale numbers. Fix direction: `useEffect(() => setTransactions(initialTransactions), [initialTransactions])`.
- **`SessionTimeout.tsx` comment vs. code disagree**: comment says "30 minutes", the actual constant `TIMEOUT_MS = 600 * 60 * 1000` is 10 hours. If a user reports "I wasn't logged out after 30 min of idling," this is why — that's expected per the code, not a bug, but the comment is misleading.
- **`transactions/import` date-parsing fallback ambiguity**: the regex fallback branch's inline comment says "assume MM/DD/YYYY (Excel default)" but the actual return expression treats the first captured group as day and the second as month (`DD/MM/YYYY`-style). Worth a real test against an actual ambiguous-format Excel file (e.g. `03/04/2026`) before trusting either the comment or the code blindly.
- **Inconsistent client-side resync-after-mutation strategy**: several "Tab" components seed local state from server props once and never resync; some of their own mutation handlers call `router.refresh()`, others only patch local state optimistically, inconsistently even within the same file:
  - `ApiKeysTab`: revoke calls `router.refresh()`, create does not.
  - `UsersTab`: create skips `router.refresh()`, update/delete both call it.
  - `OrdersTab` is the model to follow: it never copies the `transactions` prop into local state at all, so `router.refresh()` alone is always sufficient.
- **No auth on two routes**: `POST /api/sync` and `GET /api/test-export` have zero authentication. `/api/sync` is presumably meant to be called by a trusted internal job (network-level trust), and `/api/test-export` looks like a leftover dev diagnostic. Both should be reviewed before any public/internet-facing deployment — as written, anyone who discovers either URL can inject/soft-delete transaction data or trigger a full Google Sheets export.
- **`/api/sync` assumes a complete global snapshot**: it soft-deletes any `source: "provider"` row missing from the current payload, with no per-customer scoping on that diff. A sync call that only includes a subset of customers (intentionally or due to a bug upstream) will incorrectly soft-delete every other customer's provider-sourced rows.
- **Referential-integrity gap on unit deletion**: `DELETE /api/units/[id]` blocks deletion if `_count.transactions > 0`, but does not check `StockAssignment.targetDeviceNumber` — a unit referenced only via stock assignments (not direct transactions) can still be deleted, potentially orphaning those assignment rows.
- **`getStockInfo` (in `lib/chat-tools.ts`) filters in-memory instead of at the query level**: it fetches up to 30 `StockAssignment` rows unfiltered by customer, then discards non-matching ones in JS when scoped to a customer — functionally correct but inconsistent with the other three chat tools (`queryTransactions`, `getSummaryStats`, `searchParts`), which all filter via the Prisma `where` clause. Worth aligning if this function is extended.
- **`components/ui/button.tsx` (shadcn `Button`) is effectively unused** by `components/dashboard/*` — every dashboard component hand-rolls its own `<button>` with inline Tailwind and the hardcoded John Deere green (`#367C2B`/`#2d6423`) instead of using the shared primitive's `variant`/`size` props. Not a bug, but don't assume introducing a new dashboard button should use `<Button>` just because it exists — check the surrounding file's existing convention first.
- **`StatusBadge`/`STATUS_MAP`** (`components/dashboard/StatusBadge.tsx` + `data/customer.ts`) isn't referenced by any component covered during this documentation pass — confirm it's still in use (or find its actual caller) before assuming it's dead code.
- **`prisma migrate dev` fails with `P3006`/`P1014`** ("The underlying table for model `Unit` does not exist") when replaying migration `20260411000000_add_model_category_rename_invoicedate` against a fresh shadow database — discovered 2026-07-05 while adding the `Unit.customerAccount` NOT NULL constraint. This means the migration history doesn't cleanly reproduce the current schema from scratch (likely because some earlier schema change was applied via `prisma db push` rather than a tracked migration). Workaround used: `prisma db push` to apply the schema change directly, then `prisma migrate resolve --applied <name>` to mark a hand-written migration file as applied without going through the shadow-db validation. If you need to run `prisma migrate dev` for a future change and hit the same error, use the same workaround rather than fighting the shadow database — properly repairing the migration history (e.g. via `prisma migrate diff` + baselining) is a separate, larger task nobody has done yet. Also note: this project's `.env` has no `DIRECT_URL` set even though `schema.prisma` requires one; both `db push` and `migrate resolve` needed it passed in explicitly (Neon's `DATABASE_URL` here is already a non-pooled endpoint — no `-pooler` in the hostname — so reusing it as `DIRECT_URL` was safe in this case).

## Inconsistent-by-design (not necessarily worth "fixing", but know about it)

- **Response envelope shapes vary across routes**: most use `{ error: "..." }` / `{ success: true }`, but `admin/users/[id]` DELETE returns `{ ok: true }`, and `/api/chat` returns plain `Response`/`{ text }` instead of `NextResponse.json`.
- **Soft-delete vs. hard-delete split**: `Transaction` is always soft-deleted (`isDeleted`/`deletedAt`); `Unit`, `StockAssignment`, `ApiKey`, `User` are all hard-deleted. If you're adding a new deletable model, decide deliberately which pattern fits rather than copying the nearest example.
- **`PATCH /api/transactions/[id]` nulls omitted fields** (full-overwrite semantics) while **`PATCH /api/stock-assignments/[id]` only updates provided fields** (partial-update semantics). Know which one you're calling before omitting a field you meant to preserve.
- **`exportToSheets()` only mirrors `Transaction`/`StockAssignment`** — it is never triggered by `Unit`, `User`, or `ApiKey` mutations, so the Google Sheet is not a full database mirror.
- **`ImportModal.onImported()` only fires when `result.success > 0`** — a fully-failed import (all rows errored) never triggers the parent's `router.refresh()`/state update. This is usually fine (nothing changed server-side to refresh), but don't rely on `onImported` firing as a proxy for "the import request completed."

## Magic strings to grep for before renaming/removing

- `"WSL-000039232"` — hardcoded stock-placeholder `Unit.deviceNumber`, referenced in `app/api/units/[id]/route.ts` (immutable), `app/api/transactions/import/route.ts` (auto-assigned, exported as `SYSTEM_STOCK_DEVICE` in that file) and now also explicitly exempted from `validateUnitOwnership()` there. Since the 2026-07-05 backfill, its `customerAccount` is `"C1918"` purely to satisfy the NOT NULL constraint — this is **not** a real business fact (it's a shared system placeholder, not actually owned by that customer). Two consequences to remember: (1) `GET /api/units` scoping means only C1918/admin ever see it in a device picker — other customers can still get it auto-assigned via Stock-category Excel import (the exemption above), but can't pick it manually in `TransactionFormModal`/`StockAssignmentModal`; (2) if this device number is ever renamed or a second "shared bucket" unit is introduced, update the `SYSTEM_STOCK_DEVICE` constant and the ownership exemption together, in the same file.
- `"STOCK"` (device number sentinel, distinct from the above) — rejected as a `targetDeviceNumber` in `app/api/stock-assignments/route.ts` and `[id]/route.ts`; also happens to be the seeded `Unit.deviceNumber` in `prisma/seed.ts` (a third, coincidentally-named thing — don't conflate the seed unit with the sentinel check).
- Category codes `"P"`/`"R"`/`"S"` — single source of truth for the accepted words is `CATEGORY_MAP` in `app/api/transactions/import/route.ts`; every other place that reads `category` expects it to already be one of these three letters.
